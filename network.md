# Linuxネットワーク基礎（やさしめ・丁寧）— これで8割理解を目指すまとめ

対象：Linux初心者（WSL/Ubuntu でもOK）  
ゴール：**「今、ネットワークがどうなってるか」を自分で確認できて、基本トラブルを切り分けできる**ようになる

---

## 0. まず全体像（いちばん大事）
ネットワークの話は、だいたいこの順で考えると迷いにくいです。

1. **自分のPCはどのネットワークにいる？（IPアドレス）**
2. **どこに出ていく？（ルーティング／デフォルトゲートウェイ）**
3. **名前をどうやってIPに変える？（DNS）**
4. **通信の出入り口は開いてる？（ポート／ファイアウォール）**

これが分かると、ネットワークは “雰囲気” じゃなくて “確認して直すもの” になります。

---

## 1. IPアドレスとサブネット（最初の壁を一撃で超える）

### 1-1. IPアドレスとは
ネットワーク上の「住所」です。例：
- `192.168.1.10`（家庭内LANでよくある）
- `10.0.0.5`（会社や仮想ネットワークでよくある）

### 1-2. サブネット（CIDR）とは
`/24` とか `/16` とかのやつ。意味は「同じネットワークにいる範囲」。

例：
- `192.168.1.10/24`
  - `/24` ＝ 先頭24bit（だいたい `192.168.1.*` が同じ仲間）
  - つまり `192.168.1.20` には **直接** 送れる可能性が高い

ざっくり覚え：
- `/24` → 家庭LANでよくある「同じ部屋のネットワーク」感
- `/16` → もっと広い仲間（同じ建物くらいの感）

---

## 2. ネットワーク状態を確認する基本コマンド

### 2-1. インターフェース（NIC）とIPを見る：`ip`
昔は `ifconfig` が定番でしたが、今は `ip` が主流です。

- インターフェース一覧とIP：
```bash
ip addr
# 省略形
ip a
```

見るポイント：
- `eth0` / `ens33` など：有線（環境で名前が違う）
- `wlan0`：無線（環境で違う）
- `lo`：ループバック（自分自身＝localhost）

IPが付いてるかどうかで、まず「ネットワークに参加できてるか」が分かります。

---

### 2-2. ルーティング（どこへ出ていくか）：`ip route`
```bash
ip route
```

見るポイント（超重要）：
- `default via 192.168.1.1 dev eth0` のような行  
  これが **デフォルトゲートウェイ**（とりあえず外に出る玄関）

ネットが繋がらないとき：
- **default が無い** → 外へ出る道がない

---

### 2-3. DNS（名前解決）：`resolvectl` / `cat /etc/resolv.conf`
DNSは「`google.com` → `142.250...`」みたいに名前をIPに変える仕組み。

確認：
```bash
# systemd-resolved が動いてる環境
resolvectl status

# どの環境でも一応見れることが多い
cat /etc/resolv.conf
```

---

## 3. 接続チェック（疎通確認）で「どこまで通ってるか」を判定

### 3-1. `ping`（生きてる？届く？）
```bash
ping -c 4 8.8.8.8
```

- これは **IP直指定**。DNSを使わない。  
  → これが通るなら「外へ出る経路」はだいたいOK。

次にDNS確認：
```bash
ping -c 4 google.com
```
- こっちがダメで、IP直指定はOKなら  
  → **DNSが怪しい**（名前解決ができてない）

---

### 3-2. `curl`（HTTP/HTTPSで実用確認）
```bash
curl -I https://example.com
```
- `-I` はヘッダだけ表示（軽い）
- 実際のWeb通信ができているか分かる

---

### 3-3. `traceroute` / `tracepath`（途中でどこまで届く？）
```bash
tracepath 8.8.8.8
# traceroute が入っていれば
traceroute 8.8.8.8
```
途中の経路が見えるので、「どこで詰まってるか」の当たりがつきます。

---

## 4. ポートとサービス（“つながる／つながらない” の核心）

### 4-1. ポートとは
同じIPの中で、アプリごとに分ける「部屋番号」です。

例：
- 80: HTTP
- 443: HTTPS
- 22: SSH

---

### 4-2. いま待ち受けしてる（Listen）ポートを見る：`ss`
```bash
ss -tulpen
```

よく使う見方：
- `LISTEN` になってるか
- `0.0.0.0:3000` みたいなら「全IPで待ち受け」
- `127.0.0.1:3000` なら「自分からしかアクセスできない」（外部から来ない）

※ Node/Express で “外から繋がらない” の原因が `127.0.0.1` bind のこと、よくあります。

---

### 4-3. 接続できるか試す：`nc`（netcat）
```bash
nc -vz 127.0.0.1 3000
```
- そのポートに接続できるかの確認
- `succeeded` ならOK

---

## 5. よく触る設定ファイル（最小限）

### 5-1. `/etc/hosts`（DNSより先に効く “手動の名前帳”）
```bash
cat /etc/hosts
```

例：
```
127.0.0.1 localhost
```

`example.local` を自分で固定したいときに使うことがあります。

---

### 5-2. `/etc/resolv.conf`（DNSサーバ設定の参照先）
```bash
cat /etc/resolv.conf
```

ただし環境によっては自動生成なので、直接編集しない方がよいことも多いです。

---

## 6. （軽めに）ファイアウォール：`ufw`
Ubuntu系でよく使われます（環境によって入ってない場合もあります）。

状態確認：
```bash
sudo ufw status
```

許可例（例：3000番を開ける）：
```bash
sudo ufw allow 3000/tcp
```

---

## 7. トラブルシュートの“型”（これが8割の正体）

「繋がらない」を聞いたら、次を順番に潰すと速いです。

### Step 1: IPは付いてる？
```bash
ip a
```

### Step 2: 外へ出る道はある？
```bash
ip route
# default があるか
```

### Step 3: IP直指定は通る？
```bash
ping -c 4 8.8.8.8
```

### Step 4: 名前解決は通る？
```bash
ping -c 4 google.com
```

### Step 5: アプリは待ち受けしてる？
```bash
ss -tulpen
```

### Step 6: どこに bind してる？
- `127.0.0.1` なら “自分だけ”
- `0.0.0.0` なら “外からも”

### Step 7: ファイアウォールは？
```bash
sudo ufw status
```

## 8. 最低限の用語集（この章で出てくるやつ）
- **インターフェース**：ネットワーク接続口（eth0 など）
- **IPアドレス**：住所
- **サブネット**：同じ仲間の範囲（/24 など）
- **ルーティング**：どっちへ送るかの道案内
- **デフォルトゲートウェイ**：外へ出る玄関（default via …）
- **DNS**：名前 → IP 変換
- **ポート**：アプリの部屋番号
- **Listen**：待ち受け（サーバ側が待っている状態）

## 9. 手を動かすミニ演習（理解が定着しやすい）
以下を“自分の環境の値”で言えるようになると強いです。

1) 自分のIPは？
```bash
ip a
```

2) デフォルトゲートウェイは？
```bash
ip route
```

3) DNSサーバは？
```bash
cat /etc/resolv.conf
# or
resolvectl status
```

4) `8.8.8.8` と `google.com` で ping の結果がどう違う？
```bash
ping -c 2 8.8.8.8
ping -c 2 google.com
```

5) いま待ち受けしてるポートは？
```bash
ss -tulpen
```

## 10. ここまでで “8割” できている状態
次のことが説明できたら、ネットワーク基礎はかなり掴めています。

- 「IPは付いてる／付いてない」を確認できる
- 「default route がある／ない」を確認できる
- 「IP直指定はOK、名前はNG」→ DNSだと判断できる
- 「待ち受けしてる／してない」を `ss` で確認できる
- 「127.0.0.1 bind が原因」を疑える

---

## 付録：よく使うコマンド早見
```bash
ip a                 # IP/インターフェース
ip route             # ルーティング（default確認）
ping -c 4 8.8.8.8     # IP疎通
ping -c 4 google.com  # DNS含む疎通
curl -I https://example.com  # HTTP疎通
ss -tulpen            # 待ち受けポート確認
resolvectl status     # DNS確認（環境による）
cat /etc/resolv.conf  # DNS確認（環境による）
sudo ufw status       # FW確認（入っていれば）
```
